# 挂号业务前端对接文档

> 本文档为挂号业务的**唯一前端交付物**，覆盖号源放号、现场挂号、患者网上抢号、挂号查询与状态流转全部接口。
> 后端依赖：MySQL + Redis + RabbitMQ，需先 `docker compose up -d mysql redis rabbitmq`。

---

## 一、概述与角色路由（必读）

挂号业务有**两条挂号入口**，按角色路由：

| 入口 | 路径 | 谁能用 | 模式 |
|------|------|--------|------|
| **现场挂号** | `POST /api/v1/register` | ADMIN、DOCTOR | 同步落库 |
| **网上抢号** | `POST /api/v1/patient/register/grab` | PATIENT | 异步秒杀（MQ 削峰） |

> **患者可以挂号。** 患者的挂号入口是 `/api/v1/patient/register/grab`（异步抢号链路）。
> `POST /api/v1/register` 对 PATIENT 返回 **403** 是**角色路由**——现场挂号入口归 ADMIN/DOCTOR，**不是"患者不能挂号"**。

### 角色 × 能力总表

| 角色 | 放号 | 现场挂号 | 网上抢号 | 查询 / 状态流转 |
|------|:----:|:--------:|:--------:|:---------------:|
| ADMIN | ✅ | ✅（可指定医生） | ❌ | 全部挂号 |
| DOCTOR | ❌ | ✅（仅本人接诊） | ❌ | 本人接诊记录 |
| PATIENT | ❌ | ❌（403，走抢号入口） | ✅ | 本人关联记录 |

### 两入口关键差异速查

| 维度 | 现场挂号 `/api/v1/register` | 网上抢号 `/api/v1/patient/register/grab` |
|------|----------------------------|------------------------------------------|
| 鉴权 | ADMIN / DOCTOR | PATIENT |
| 模式 | 同步，立即返回完整挂号 | 异步，返回票据 → 轮询拿结果 |
| `visitDate` 格式 | `LocalDateTime`（`yyyy-MM-ddTHH:mm:ss`） | `LocalDate`（`yyyy-MM-dd`） |
| 必填字段 | realName/gender/deptmentId/registLevelId/settleCategoryId | employeeId/visitDate/noon/registLevelId/settleCategoryId |
| 患者身份 | 由 cardNumber+realName+gender+birthdate 匹配自动建 link | 从登录态取，**不收 patientId** |
| 号源扣减 | 同步扣 DB（满号→409，未放号放行） | Redis Lua 原子扣 + MQ + DB 兜底 |
| 病历号 | 服务端生成，响应直接返回 | SUCCESS 后从票据查询接口取 |

## 二、字典下拉接口（三角色共用）

现场挂号表单与患者抢号页的四个下拉（科室 / 医生 / 挂号级别 / 结算类别）都走下列只读接口。这四个端点已对 **ADMIN / DOCTOR / PATIENT 三角色放开**（写操作仍仅 ADMIN）。

| 下拉 | 接口 | 角色 | 说明 |
|------|------|:----:|------|
| 科室 | `GET /api/v1/department/all` | ADMIN/DOCTOR/PATIENT | 全部有效科室，不分页 |
| 医生（按科室联动） | `GET /api/v1/employee/dept/{deptId}` | ADMIN/DOCTOR/PATIENT | 指定科室下的全部有效医生；选完科室后调此接口填医生下拉 |
| 挂号级别 | `GET /api/v1/regist-level/all` | ADMIN/DOCTOR/PATIENT | 全部有效级别，含 `registFee`（可用于费用预览） |
| 结算类别 | `GET /api/v1/settle-category/all` | ADMIN/DOCTOR/PATIENT | 全部有效结算类别 |

响应均为 `Result<List<...Response>>`，字段同各管理端的 Response DTO（如 `DepartmentResponse{id, deptCode, deptName, ...}`、`EmployeeResponse{id, realname, deptmentId, deptName, registLevelId, registLevelName, ...}`）。

> 联动顺序建议：先加载科室 + 级别 + 结算类别（三个 `/all` 并行），用户选科室后再调 `GET /employee/dept/{deptId}` 填医生下拉。
> 注意：这四个接口的分页 `GET /`、详情 `GET /{id}` 仍仅 ADMIN；患者/医生只用上表的 `/all` 与 `/dept/{deptId}`。

## 三、号源模型与 ADMIN 放号

每个医生每天的号数固定，由 ADMIN 提前「放号」。号源表 `doctor_daily_quota(employee_id, quota_date, noon, capacity, remaining)`。

> **抢号 / 现场挂号前必须先放号。** 未放号时：患者抢号 → 404「该医生当日未放号」；现场挂号 → 不限号放行（向后兼容）。

### `POST /api/v1/register/quota`（仅 ADMIN）

放号 / 重置号源。请求体：
```json
{ "employeeId": 1, "quotaDate": "2026-07-03", "noon": "上午", "capacity": 50 }
```
| 字段 | 必填 | 说明 |
|------|:----:|------|
| `employeeId` | 是 | 医生 ID |
| `quotaDate` | 是 | 放号日期 `yyyy-MM-dd` |
| `noon` | 是 | `上午` / `下午` |
| `capacity` | 是 | 总号源数 |

- **upsert**：同一医生+日期+午别重复放号会**重置 remaining=capacity**（覆盖剩余）。
- 放号同时把 `remaining` 灌入 Redis 库存 key `regist:stock:{emp}:{date}:{noon}`。
- 响应：`Result<QuotaResponse>`，含 `id/employeeId/quotaDate/noon/capacity/remaining`。

### `GET /api/v1/register/quota?employeeId=1&date=2026-07-03`（仅 ADMIN）

返回该医生当日所有午别号源列表（含 `remaining`），供前端展示「剩余号源」。

> 患者侧目前无只读号源查询接口；患者抢号页直接选医生/日期/午别提交，约满由后端 409 反馈。如需患者侧剩余号源展示，后续再加只读接口。

## 四、现场挂号 `POST /api/v1/register`（ADMIN / DOCTOR）

### 3.1 v1.4 加固要点（对前端的影响）

| # | 加固点 | 旧契约 | 新契约 | 前端需调整 |
|---|--------|--------|--------|-----------|
| 1 | 建号即置 `visit_state=1` | 客户端传，常落 NULL | 服务端强制 `1`（已挂号） | 无需传 `visitState` |
| 2 | `case_number` 服务端生成且全局唯一 | 客户端必传 `caseNumber` | **不再发送**，服务端按 `BL+yyyyMMddHHmmss+2位随机` 生成 | 移除"病历号"输入框，建号后从响应取 |
| 3 | `regist_money` 按级别推导 | 客户端传 `registMoney` | **不再发送**，取 `regist_level.regist_fee` | 移除"挂号费"输入；选级别后可调级别接口预览费用 |
| 4 | 创建流程原子化 | 挂号与患者关联分两步 | 单事务，全成或全败 | 失败时不会出现"半条"挂号 |

> **已移除字段**：`caseNumber`、`registMoney`、`visitState` —— 不要再发送。Jackson 默认忽略未知字段，多发不会报错但**直接丢弃不生效**，建议尽快移除避免误以为传值生效。

### 3.2 鉴权与医生归属

- `@RequireRole({ADMIN, DOCTOR})`，PATIENT 调用 → **403**（走抢号入口）。
- **DOCTOR**：请求体里的 `employeeId` 会被服务端**强制覆盖**为当前登录医生，前端无需传（传了也无效）。
- **ADMIN**：可通过 `employeeId` 指定接诊医生；不传则 `employee_id` 为空（待补）。
- **号源**：已放号则同步扣（满号→409），未放号放行。

### 3.3 请求体（JSON）

| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `realName` | string | 是 | 患者真实姓名 |
| `gender` | string | 是 | `男` / `女` |
| `cardNumber` | string | 否 | 身份证号（18 位）。若与某 `patient` 的 `card_number+real_name+gender+birthdate` 严格一致，服务端自动建立患者↔挂号关联 |
| `birthdate` | string(yyyy-MM-dd) | 否 | 出生日期（自动关联时需与 patient 一致） |
| `age` | int | 否 | 年龄 |
| `ageType` | string | 否 | `岁` / `月` / `天` |
| `homeAddress` | string | 否 | 家庭住址 |
| `visitDate` | string(yyyy-MM-ddTHH:mm:ss) | 否 | 就诊日期时间 |
| `noon` | string | 否 | `上午` / `下午` |
| `deptmentId` | int | 是 | 挂号科室 ID |
| `employeeId` | int | 否 | 接诊医生 ID。**DOCTOR 传无效**（强制覆盖为本人）；仅 ADMIN 生效 |
| `registLevelId` | int | 是 | 挂号级别 ID（决定 `registMoney`，不存在则 404） |
| `settleCategoryId` | int | 是 | 结算类别 ID |
| `isBook` | string | 否 | 是否预约：`是` / `否` |
| `registMethod` | string | 否 | 挂号方式：`现金`/`银行卡`/`医保卡`/`微信`/`支付宝` |

### 3.5 错误码

| 场景 | HTTP | `code` | `message` |
|------|:----:|:------:|-----------|
| 成功 | 200 | 200 | 操作成功 |
| PATIENT 调用 | 403 | 403 | 权限不足 |
| `@Valid` 校验失败 | 200 | 400 | 参数错误（含字段明细） |
| `registLevelId` 不存在 | 404 | 404 | 挂号级别不存在: \<id\> |
| 号源已约满（已放号时） | 409 | 409 | 已约满 |
| 病历号生成冲突（极罕见） | 409 | 409 | 病历号生成冲突，请重试 |
| 未登录 / token 失效 | 401 | 401 | 未登录 / 令牌已过期… |

> 409 病历号冲突几乎不会发生（同秒同随机数碰撞且重试 3 次仍冲突），前端可提示"网络繁忙，请重试"。

## 五、网上抢号（PATIENT，异步秒杀）

### 4.1 异步链路

```
患者 POST /api/v1/patient/register/grab
  → Redis Lua 原子扣减 regist:stock:{emp}:{date}:{noon}
     返回 -1=未放号 / 0=约满 / 1=成功
  → 成功则投递 RabbitMQ，立即返回 PENDING 票据
  → 消费者 MySQL 兜底 UPDATE...WHERE remaining>0 + 落库挂号
  → 票据变 SUCCESS（含 registerId/caseNumber）或 FAILED（约满/系统异常）
患者 GET /api/v1/patient/register/grab/{ticketNo} 轮询结果
```

- **幂等 / 一人一号**：同一患者每半日（上午/下午）仅一号。重复提交返回既有票据；FAILED/CANCELLED 后可重抢。
- 患者身份由 Service 从登录态取，**Controller 不收 `patientId`**。

### 4.2 `POST /api/v1/patient/register/grab`

请求体：
```json
{
  "employeeId": 1,
  "visitDate": "2026-07-03",
  "noon": "上午",
  "registLevelId": 1,
  "settleCategoryId": 1
}
```
| 字段 | 类型 | 必填 | 说明 |
|------|------|:----:|------|
| `employeeId` | int | 是 | 抢指定医生 |
| `visitDate` | string(yyyy-MM-dd) | 是 | 就诊日期（**注意是 LocalDate，不含时分秒**） |
| `noon` | string | 是 | `上午` / `下午` |
| `registLevelId` | int | 是 | 决定挂号费 |
| `settleCategoryId` | int | 是 | 结算类别 |

响应：`Result<RegistrationGrabResponse>`
```json
{ "ticketNo": "QH2026070309301247", "status": "PENDING" }
```
- `status=PENDING`：已入队，前端开始轮询。
- `status=SUCCESS`：极快落库时直接成功（幂等返回既有）。
- `status=FAILED`：之前该票据已失败（幂等返回）。

错误码：

| HTTP | `code` | 含义 |
|:----:|:------:|------|
| 404 | 404 | 该医生当日未放号 |
| 409 | 409 | 该号源已约满 |
| 409 | 409 | 请勿重复抢号（一人一号，半日内已有进行中/成功号）|
| 400 | 400 | 参数校验失败 |
| 403 | 403 | 非 PATIENT 或未绑定患者档案 |

> 「重复抢号」通常不会返回 409——幂等逻辑会直接返回既有 PENDING/SUCCESS 票据；409 仅在并发双击的极端竞态下出现。

### 4.3 `GET /api/v1/patient/register/grab/{ticketNo}`

响应：`Result<RegistrationTicketResponse>`
```json
{
  "ticketNo": "QH2026070309301247",
  "employeeId": 1,
  "visitDate": "2026-07-03",
  "noon": "上午",
  "status": "SUCCESS",
  "registerId": 11,
  "caseNumber": "BL2026070309301247",
  "failReason": null,
  "createTime": "2026-07-03T09:30:12"
}
```
- `status=PENDING`：继续轮询。
- `status=SUCCESS`：抢号成功，`registerId`/`caseNumber` 可用（可跳转就诊详情）。
- `status=FAILED`：失败，`failReason` 含原因（已约满/未放号/系统异常），可换号源重抢。
- 非本人票据 → 404。

### 4.4 状态机

```
POST /grab ──> PENDING ──消费成功──> SUCCESS (registerId/caseNumber 回填)
                │
                └──消费失败──> FAILED (failReason: 已约满/未放号/系统异常)
                                  │
                                  └── 重新 POST /grab（同半日）→ 删旧建新 PENDING
```

### 4.5 前端轮询建议

- 拿到 `ticketNo` 后每 **1–1.5s** 调一次 `GET /grab/{ticketNo}`。
- 超时 **~15–30s** 仍 PENDING → 提示「排队中，请稍后在就诊记录查看」并停止轮询。
- SUCCESS → 提示成功 + 跳转就诊详情；FAILED → 提示原因，允许重选。
- **防重复点击**：提交按钮点击后立即禁用，直到拿到 `ticketNo`；服务端已幂等，双击安全。

## 六、挂号查询与状态流转（三角色共用）

| HTTP | Path | 角色 | 说明 |
|------|------|------|------|
| GET | `/api/v1/register?pageNum&pageSize&caseNumber&realName&visitState&visitDateStart&visitDateEnd&deptmentId` | ADMIN/DOCTOR/PATIENT | 分页列表。DOCTOR 仅本人接诊；PATIENT 仅本人关联；ADMIN 全部 |
| GET | `/api/v1/register/case/{caseNumber}` | 同上 | 按病历号精确查（归属校验，非本人 → 404） |
| GET | `/api/v1/register/{id}` | 同上 | 按 ID 查（归属校验） |
| PUT | `/api/v1/register` | ADMIN/DOCTOR | 修改挂号信息（DOCTOR 仅本人接诊） |
| PUT | `/api/v1/register/{id}/state?visitState=` | ADMIN/DOCTOR | 更新看诊状态 |

> 列表/详情接口的 `caseNumber`、`registMoney`、`visitState` 字段含义同第四节。

### `visit_state` 生命周期

```
建号 ──> 1 已挂号 ──(医生接诊)──> 2 医生接诊 ──(看诊结束)──> 3 看诊结束
              └──(退号)──> 4 已退号
```

- 建号后固定为 `1`（现场挂号响应直接返回 1；抢号 SUCCESS 后落库的挂号也是 1），前端**无需**也**不能**在建号时指定。
- 状态流转由 `PUT /api/v1/register/{id}/state?visitState=<n>` 驱动（DOCTOR 仅限本人接诊的挂号；ADMIN 任意）。

## 七、前端改造与对接清单（按角色）

### ADMIN
1. **放号页**：选医生 + 日期 + 午别 + 容量 → `POST /quota`；列表展示 `GET /quota`。
2. **现场挂号表单**：移除"病历号"输入框（自动生成）、"挂号费用"输入框（按级别自动算）；可选接诊医生。
3. **级别联动**：选择挂号级别后，可调 `GET /api/v1/regist-level` 取 `registFee` 给用户预览费用（仅为展示，最终以建号响应的 `registMoney` 为准）。
4. **建号成功后**：从响应 `data.caseNumber` 取出生成的病历号展示；用 `data.id` 直接跳转详情页，无需二次查询列表。
5. **列表/详情**：用第六节速查接口。

### DOCTOR
1. **现场挂号表单**：隐藏"医生"字段（强制为本人，无法选择）；其余同 ADMIN。
2. **状态流转**：接诊时 `PUT /{id}/state?visitState=2`，看诊结束 `=3`，退号 `=4`。
3. **列表**：仅返回本人接诊记录。

### PATIENT
1. **抢号页下拉**：四个下拉用第二节字典接口——`GET /department/all` + `GET /regist-level/all` + `GET /settle-category/all` 并行加载，选科室后调 `GET /employee/dept/{deptId}` 联动医生。
2. **抢号页**：选医生 + 日期 + 午别 + 级别 + 结算类别 → `POST /grab` 拿 `ticketNo` → 轮询 `GET /grab/{ticketNo}`。
3. SUCCESS → 提示成功 + 跳转就诊详情；FAILED → 提示原因，允许重选换号源重抢。
4. **就诊记录**：抢号成功后，`GET /api/v1/patient/records` 会自动包含该挂号（经 patient_register_link 桥接）。
5. **防重复点击**：提交后立即禁用按钮，直到拿到 `ticketNo`。

### 通用
- 错误处理：404（级别不存在/未放号/票据非本人）→ 相应提示；409（约满/病历号冲突）→ 提示重试。
- 旧字段清理：现场挂号请求体停止发送 `caseNumber`/`registMoney`/`visitState`。

## 八、依赖与运维

- **RabbitMQ**：`docker compose up -d rabbitmq`；管理台 http://localhost:15672（admin/admin2024）。队列 `regist.grab.queue`，死信 `regist.dlx.queue`（消费失败经重试后进入，供人工排查）。
- **Redis**：库存 key `regist:stock:{emp}:{date}:{noon}`，Lua 原子扣减；消费失败/退号自动 `INCR` 回补。
- **MySQL 兜底**：`doctor_daily_quota.remaining` 是权威库存，`UPDATE ... WHERE remaining>0` affected rows=0 即满号。Redis 与 DB 偶发漂移由消费端兜底 + refund 修正，极端情况可重新放号 `POST /quota` 重 seed。
