# 前端登录 / 注册对接文档（统一账号化）

> 本文档面向前端，描述 **患者(PATIENT) / 医生(DOCTOR) / 管理员(ADMIN)** 三角色的登录与注册（建号）逻辑及全部涉及接口。
> 后端登录链路已统一从 `user_account` 表查询，三角色共用同一登录入口；JWT v2 账号化认证，三角色平级，权限由后端 `@RequireRole` 强制校验，前端只做体验控制。

---

## 一、总体模型

### 1.1 统一账号表 `user_account`

所有角色共用一张账号表，登录入口统一。账号与业务身份通过外键关联：

| 角色 | 关联身份 | 账号字段 |
|------|----------|----------|
| `PATIENT`  | `patient.id`     | `patient_id`  |
| `DOCTOR`   | `employee.id`    | `employee_id` |
| `ADMIN`    | `employee.id`（可空） | `employee_id` |

账号字段：`username`(唯一)、`password`(BCrypt)、`role`、`status`(1启用/0禁用)、`token_version`、`delmark`。

- 密码：明文经 HTTP 传输，后端 **BCrypt** 哈希入库；旧 MD5 密码在首次登录时自动升级为 BCrypt。
- 三角色**平级、无继承**：Controller 显式列出允许角色，前端不可凭角色推断越权接口。

### 1.2 三种"注册"路径（重点）

本系统**没有统一 `/register` 接口**，三角色建号方式不同：

| 角色 | 建号方式 | 接口 | 是否需登录 |
|------|----------|------|-----------|
| PATIENT | **自助注册**（公开） | `POST /api/v1/auth/patient/register` | 否（`@Public`） |
| DOCTOR  | **管理员建号** | `POST /api/v1/account/employee` | 是（仅 ADMIN） |
| ADMIN   | **管理员建号** | `POST /api/v1/account/employee` | 是（仅 ADMIN） |

> DOCTOR / ADMIN 不能自助注册，必须由已登录 ADMIN 在「账号管理」页创建并绑定一个已存在的 `employee`。
> 存量员工账号由迁移脚本 `sql/v2.1_employee_account_migration.sql` 从 `employee` 表一次性迁入（工号即账号）。
> 初始 ADMIN 由数据库初始化脚本预置。

### 1.3 登录入口（三角色统一）

```
POST /api/v1/auth/login   ← @Public，无需 Token
```

后端依据 `user_account.role` 派发身份，前端依据响应 `role` 做路由分流。

## 二、登录逻辑

### 2.1 接口

```
POST /api/v1/auth/login
Content-Type: application/json
```

**请求体：**

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| `username` | String | 是 | 登录账号（`user_account.username`，迁移后的员工即原工号字符串） |
| `password` | String | 是 | 密码明文 |

```json
{ "username": "doctor01", "password": "123456" }
```

> 迁移自 `employee` 的账号，`username` = 工号字符串（如工号 `1` → 用户名 `"1"`）。

### 2.2 后端流程

1. **锁定校验**：同一 `username` 连续失败 ≥5 次 → 锁定 15 分钟（Redis `login:fail:{username}`），返回 423。
2. **查账号**：按 `username` 查 `user_account`；不存在/已逻辑删除 → 统一返回"账号或密码错误"。
3. **启用校验**：`status != 1` → 同样统一安全失败。
4. **密码校验**：支持 BCrypt 与旧 MD5；旧 MD5 校验通过后**立即升级为 BCrypt**。
5. **角色关联校验**：PATIENT 须有合法 `patient_id` 且 `patient` 存在；DOCTOR 须有合法 `employee_id` 且 `employee` 存在；ADMIN 的 `employee_id` 可空。不合法 → 拒绝登录。
6. **签发 JWT v2**：清除失败计数，返回 Token + 过期时间 + 用户信息。

### 2.3 成功响应（HTTP 200）

```json
{
  "code": 200,
  "message": "操作成功",
  "data": {
    "token": "eyJhbGciOiJIUzI1NiJ9...",
    "expiresAt": 1782216675000,
    "userInfo": {
      "accountId": 2,
      "role": "DOCTOR",
      "employeeId": 1,
      "patientId": null,
      "id": 1,
      "realname": "测试医生",
      "deptmentId": 1,
      "deptName": "神经内科",
      "registLevelId": 1,
      "registLevelName": "普通号"
    }
  }
}
```

| 字段 | 说明 |
|------|------|
| `data.token` | 裸 JWT，前端持久化；请求受保护接口时拼 `Authorization: Bearer <token>` |
| `data.expiresAt` | 过期毫秒时间戳，默认 8 小时 |
| `userInfo.role` | **`ADMIN` / `DOCTOR` / `PATIENT`**，前端路由/菜单据此渲染 |
| `userInfo.accountId` | 账号 ID |
| `userInfo.employeeId` | DOCTOR 有值；PATIENT 为 `null`；ADMIN 可空 |
| `userInfo.patientId` | PATIENT 有值；其余为 `null` |
| `userInfo.id` | 旧字段，等价 `employeeId`（PATIENT/无关联员工的 ADMIN 为 `null`） |
| `userInfo.realname` | 展示姓名（PATIENT 取 `patient.real_name`，其余取 `employee.realname`） |
| `userInfo.deptName` / `registLevelName` | DOCTOR 的科室/挂号级别；PATIENT 为 `null` |

> PATIENT 的 `userInfo` **不含**身份证号、手机号、家庭住址等敏感字段。

### 2.4 失败响应

| HTTP | code | message | 触发场景 |
|------|------|---------|----------|
| 401 | 401 | 账号或密码错误 | 用户名不存在 / 密码错 / 账号已禁用（统一安全消息） |
| 423 | 423 | 账号已锁定，请稍后再试 | 连续失败 ≥5 次，锁定 15 分钟 |

> 鉴权类异常映射为真实 HTTP 状态码（401/403/423），其余业务异常为 HTTP 200 + `Result.code`。

## 三、PATIENT 自助注册

### 3.1 接口

```
POST /api/v1/auth/patient/register   ← @Public，无需 Token
Content-Type: application/json
```

### 3.2 请求体

```json
{
  "username": "new_patient",
  "password": "mypass123",
  "realName": "新患者",
  "gender": "男",
  "birthdate": "1990-01-01",
  "cardNumber": "210102199001011234",
  "phone": "13800000000",
  "homeAddress": "沈阳市和平区"
}
```

| 字段 | 必填 | 校验规则 | 说明 |
|------|------|----------|------|
| `username` | 是 | 3-64 位，`^[a-zA-Z0-9_]+$` | 登录账号，唯一 |
| `password` | 是 | 6-72 位 | 明文，后端 BCrypt 入库 |
| `realName` | 是 | 非空 | 真实姓名 |
| `gender` | 是 | 男/女 | 性别 |
| `birthdate` | 是 | `YYYY-MM-DD` | 出生日期 |
| `cardNumber` | 是 | `^[0-9]{17}[0-9Xx]$` | 18 位身份证**仅格式校验**，非真实身份核验 |
| `phone` | 否 | — | 手机号 |
| `homeAddress` | 否 | — | 家庭住址 |

### 3.3 后端流程（事务原子）

1. 按 `cardNumber` 查 `patient`（v2.0 已从 `register` 回填）：
   - **不存在**：新建 `patient` → 新建 `PATIENT` 账号绑定该患者。
   - **已存在**：仅当 `realName / gender / birthdate` 三项**完全一致**才允许绑定；资料不一致、已绑定账号、`username` 已占用 → 统一安全失败。
2. 密码 BCrypt 入库，`status=1`、`token_version=1`。
3. 并发冲突（唯一键碰撞）兜底 → 统一安全失败，事务回滚无部分写入。

### 3.4 响应

- **成功**：HTTP 200 + `code=200`，`data` 为 `null`，**不返回 JWT**。前端提示"注册成功，请登录"并跳转登录页。
- **失败**：HTTP 200 + `code=400`，`message` 为统一安全提示：

```
注册失败，请核对提交信息或联系管理员
```

> 后端**不泄露具体原因**（身份证是否已存在、用户名是否占用等），前端只展示该统一提示。参数格式校验失败则返回 `code=400` + 字段级 `message`。

## 四、DOCTOR / ADMIN 建号（仅 ADMIN）

DOCTOR 与 ADMIN 不能自助注册，由已登录 ADMIN 通过账号管理接口创建，**必须绑定一个已存在且未绑定其他账号的 `employee`**。

### 4.1 创建 DOCTOR / ADMIN 账号

```
POST /api/v1/account/employee   ← 需要 Bearer Token，仅 ADMIN
Content-Type: application/json
```

```json
{
  "username": "doctor02",
  "password": "123456",
  "role": "DOCTOR",
  "employeeId": 2
}
```

| 字段 | 必填 | 校验规则 | 说明 |
|------|------|----------|------|
| `username` | 是 | 3-64 位，`^[a-zA-Z0-9_]+$` | 登录账号，唯一 |
| `password` | 是 | 6-72 位 | 明文，BCrypt 入库 |
| `role` | 是 | 仅 `ADMIN` / `DOCTOR`（禁止 `PATIENT`） | 角色 |
| `employeeId` | 是 | 必须存在且未绑定其他账号 | 绑定的员工 ID |

- **成功**：HTTP 200 + `code=200`，`data` 为新建账号信息（**不含密码**）。
- **失败**：HTTP 200 + `code=400/409`，`message` 描述冲突原因（员工不存在、已绑定账号、用户名重复等）。
- **权限**：非 ADMIN 调用 → HTTP 403。

### 4.2 旁路：为已存在患者补建账号

```
POST /api/v1/account/patient   ← 仅 ADMIN
```

请求体：`{ "username", "password", "patientId" }`，绑定已存在且未绑定的 `patient`。用于「管理员为患者补建账号」场景，与患者自助注册互补。

## 五、登录后通用接口（三角色共用）

| 方法 | 路径 | 鉴权 | 说明 |
|------|------|------|------|
| `GET`  | `/api/v1/auth/me`       | Bearer，三角色 | 当前用户信息（返回同登录 `userInfo`，无 token/expiresAt）。刷新页面恢复会话用 |
| `POST` | `/api/v1/auth/logout`   | Bearer，三角色 | 将当前 token 加入 Redis 黑名单，立即失效。无请求体 |
| `PUT`  | `/api/v1/auth/password` | Bearer，三角色 | 改密码：`{ "oldPassword", "newPassword"(≥6位) }`。成功后 `token_version+1`，**当前及所有历史 Token 立即 401**，需重新登录 |

## 六、账号管理 API（仅 ADMIN）

| 方法 | 路径 | 说明 |
|------|------|------|
| `GET`   | `/api/v1/account` | 分页查询（role/status/username 筛选） |
| `GET`   | `/api/v1/account/{id}` | 详情（无密码/身份证/手机号/地址） |
| `POST`  | `/api/v1/account/employee` | 创建 ADMIN/DOCTOR 账号，绑定 employee |
| `POST`  | `/api/v1/account/patient` | 创建 PATIENT 账号，绑定 patient |
| `PATCH` | `/api/v1/account/{id}/status` | 启用/禁用（body: `{"status": 0|1}`） |
| `POST`  | `/api/v1/account/{id}/reset-password` | 重置密码（body: `{"newPassword": "..."}`） |

> 禁用/重置密码 → 目标账号 `token_version+1`，所有 Token 立即 401。禁止禁用自己或最后一个 ADMIN。
> 创建账号响应不含密码及患者敏感字段。

## 七、Token 与鉴权约定（前端必读）

### 7.1 Token 传输

- 后端签发**裸 JWT**，前端存本地，请求时拼接：
  ```http
  Authorization: Bearer <裸token>
  ```
- 请求头名固定 `Authorization`，值以 `Bearer `（含空格）开头。
- JWT claims 含 `ver=2`、`role`、`tv`(token_version)、`employeeId`、`patientId`、`realname`，均由后端签名，**前端不得解析、修改或自行构造**。

### 7.2 角色来源

```
role 只能来自后端登录响应 data.userInfo.role 或 GET /api/v1/auth/me。
前端不得从下拉框、localStorage 手工设置或 URL 参数读取角色。
X-Role 等请求头无权限效力，后端不读取。
```

### 7.3 token_version 全局失效

管理员禁用账号、重置密码、用户自行改密码后，该账号 `token_version` 递增，**所有历史 Token 立即 401**。
→ 前端收到 401 必须**清空本地 token + userInfo 并跳转登录页**，不得尝试重试或复用旧 token。

### 7.4 HTTP 状态码处理（401 vs 403 必须区分）

| HTTP | Result.code | 含义 | 前端处理 |
|------|------------|------|----------|
| 200  | 200 | 正常 | 正常处理 |
| 200  | 400 | 参数/业务校验错误 | 读 `message` 展示 |
| 401  | 401 | 未登录 / Token 过期 / 无效 / tv 不匹配 | **清空 token + userInfo，跳登录页** |
| 403  | 403 | 权限不足（角色不允许） | 提示"无权访问"，**不清 token、不跳登录** |
| 423  | 423 | 账号锁定 | 提示"账号已锁定，请稍后再试" |

> 401 = 凭证问题 → 重新登录；403 = 当前角色无权限 → 仅提示。两者混用会导致登录死循环。

### 7.5 响应拦截器示例

```js
http.interceptors.response.use(
  res => {
    const { code, message, data } = res.data
    if (res.status === 401 || code === 401) {
      localStorage.removeItem('token')
      localStorage.removeItem('userInfo')
      router.push('/login')
      return Promise.reject(new Error(message || '登录已失效'))
    }
    if (res.status === 403 || code === 403) {
      return Promise.reject(new Error(message || '权限不足'))
    }
    if (code !== 200) return Promise.reject(new Error(message))
    return data
  },
  err => {
    if (err.response?.status === 401) {
      localStorage.removeItem('token')
      localStorage.removeItem('userInfo')
      router.push('/login')
    }
    return Promise.reject(err)
  }
)
```

## 八、三角色登录分流伪代码

```js
const { token, userInfo } = res.data
localStorage.setItem('token', token)
localStorage.setItem('userInfo', JSON.stringify(userInfo))

switch (userInfo.role) {
  case 'ADMIN':   router.push('/admin');   break
  case 'DOCTOR':  router.push('/doctor');  break
  case 'PATIENT': router.push('/patient'); break
  default: /* 非法角色，清 token 跳登录 */
}
```

## 十、对接 Checklist

- [ ] 登录页：用户名 + 密码 → `POST /auth/login`，按 `role` 分流路由
- [ ] 患者注册页 → `POST /auth/patient/register`，成功后跳登录页（不自动登录）
- [ ] 管理员建号页 → `POST /account/employee`（DOCTOR/ADMIN），可选 `POST /account/patient`
- [ ] 全局请求头注入 `Authorization: Bearer <token>`
- [ ] 响应拦截器区分 401（清 token 跳登录）与 403（仅提示）
- [ ] 刷新页面用 `GET /auth/me` 恢复会话
- [ ] 退出登录调 `POST /auth/logout` 后清本地 token
- [ ] 改密码 `PUT /auth/password` 成功后强制跳登录页
